Distributed State Management Engine

152 points | by sysarch [Creator] | 3 hours ago | github.com

I've built a new lightweight state engine for distributed micro-frontends. It uses a bespoke pub-sub model over WebSockets to sync state across isolated browser contexts without relying on a shared DOM or local storage locks.

Looking for feedback specifically on the conflict resolution strategy in the `sync.js` module. It currently uses a naive last-write-wins approach based on timestamp, which I know is flawed for high-concurrency offline scenarios.

netweaver [Pro] (4,210) 2 hours ago [-] [Architecture]

For your conflict resolution, you really need to look into CRDTs (Conflict-free Replicated Data Types). Last-write-wins is going to cause data loss if you have two clients mutating the same object tree simultaneously during a network partition.

I recommend checking out Yjs or Automerge. Even if you don't use them, their documentation explains the concepts well.

I reviewed `sync.js`. Lines 45-58 where you're parsing the incoming message payload: you are not catching exceptions if the JSON is malformed. If a bad message hits the socket, it will crash the engine instance entirely.

try {
  const payload = JSON.parse(msg.data);
  processSync(payload);
} catch (e) {
  console.error("Invalid sync payload", e);
}
