# Vibe Code Workspace 기획 초안

## 1. 한 줄 설명

AI로 만든 앱과 실험작이 쌓이고 방치되지 않게, 한곳에서 정리하고 반드시 피드백 받아 다음 버전으로 밀어주는 바이브 코딩 워크스페이스.

## 2. 문제 정의

바이브 코딩 도구가 좋아지면서 사람들은 앱, 랜딩 페이지, 자동화 도구, 게임, 실험 서비스를 빠르게 많이 만든다. 하지만 만든 결과물은 GitHub, Vercel, Lovable, Replit, Threads, X, Notion, 로컬 폴더에 흩어진다.

주요 문제는 다음과 같다.

- 만든 프로젝트가 많아질수록 무엇을 만들었는지 기억하기 어렵다.
- 어떤 프로젝트를 계속 밀고, 어떤 프로젝트를 접어야 할지 판단하기 어렵다.
- Threads, 긱뉴스, Reddit에 올려도 피드백을 반드시 받는다는 보장이 없다.
- 받은 피드백이 프로젝트 히스토리와 연결되지 않고 흘러간다.
- v0에서 v1, v2로 어떻게 개선됐는지 기록이 남지 않는다.
- AI로 만든 프로젝트의 데모 링크, 보안 상태, 모바일 품질, 실제 사용 가능 여부가 불명확하다.

## 3. 타깃 사용자

### 1차 타깃

- AI 코딩 도구로 사이드 프로젝트를 자주 만드는 개인
- Cursor, Claude Code, Codex, Lovable, Replit, Bolt, v0 등을 쓰는 메이커
- 만든 것은 많지만 정리, 선택, 피드백, 출시가 잘 안 되는 사람

### 2차 타깃

- 바이브 코딩 결과물을 구경하고 피드백 주는 것을 좋아하는 개발자/디자이너/창업자
- 초기 제품을 빠르게 검증하고 싶은 인디 해커
- AI 빌드 과정을 공개 포트폴리오로 남기고 싶은 사람

## 4. 핵심 포지셔닝

Product Hunt는 출시 중심이고, Threads/긱뉴스는 일회성 노출 중심이다. 이 서비스는 출시 이전과 이후를 모두 포함해 프로젝트를 관리하고 개선하는 공간이다.

포지셔닝:

> Product Hunt for launch가 아니라, AI로 만든 프로젝트들을 위한 workspace, archive, feedback queue.

## 5. 핵심 가치 제안

### 사용자 입장

- 내가 만든 AI 프로젝트를 한 화면에서 관리할 수 있다.
- 어떤 프로젝트가 진행 중이고, 어떤 프로젝트가 방치됐는지 볼 수 있다.
- 피드백을 요청하면 정해진 시간 안에 최소 피드백을 받을 수 있다.
- 받은 피드백과 반영 내역이 프로젝트별로 쌓인다.
- 결과적으로 “많이 만들기”에서 “하나씩 완성하기”로 이동할 수 있다.

### 피드백 제공자 입장

- 흥미로운 AI 프로젝트를 빠르게 써볼 수 있다.
- 유용한 피드백을 주면 크레딧, 평판, 뱃지를 얻는다.
- 디자이너, 개발자, 창업자, 일반 사용자 관점의 리뷰어 정체성을 만들 수 있다.

## 7. 핵심 화면

### 7.1 개인 워크스페이스

사용자가 가장 자주 보는 화면이다. 게시판 피드가 아니라 프로젝트 관리 보드가 첫 화면이어야 한다.

상태 컬럼:

- Ideas
- Building
- Needs Feedback
- Iterating
- Shipped
- Parked
- Archived

프로젝트 카드 정보:

- 프로젝트 이름
- 한 줄 설명

- 데모 링크 상태
- 최근 업데이트일
- 받은 피드백 수
- 반영한 피드백 수
- 사용 도구
- 다음 액션

### 7.2 프로젝트 상세

프로젝트의 현재 상태와 히스토리를 보여준다.

포함 정보:

- 제목
- 한 줄 소개
- 긴 설명
- 데모 링크
- GitHub 링크
- 스크린샷 또는 영상
- 사용한 AI 도구
- 만든 기간
- 현재 상태
- 받고 싶은 피드백 유형
- 피드백 목록
- 개선 로그
- 버전별 변경 사항

### 7.3 피드백 큐

피드백을 요청한 프로젝트들이 모이는 공간이다.

정렬 기준:

- 마감 임박
- 피드백 부족
- 내 전문 분야와 맞음
- 최신 요청

피드백 유형:

- 첫인상
- UX/UI
- 버그
- 모바일 사용성
- 기능 아이디어
- 비즈니스 가능성
- 코드/구조
- 보안/데이터 위험

### 7.4 공개 프로필

사용자의 AI 프로젝트 포트폴리오이자 아카이브다.

보여줄 정보:

- 공개 프로젝트 목록
- 현재 밀고 있는 프로젝트
- 출시한 프로젝트
- 접은 프로젝트
- 받은 피드백 수
- 반영한 피드백 수
- 제공한 피드백 수
- 주로 쓰는 AI 도구

## 8. 피드백 보장 메커니즘

서비스의 핵심 차별점은 “반드시 피드백을 받는다”는 약속이다.

### 기본 구조

- 사용자는 다른 프로젝트에 피드백을 남기면 크레딧을 얻는다.
- 크레딧을 사용해 자기 프로젝트를 피드백 큐에 올린다.
- 큐에 올라간 프로젝트는 정해진 시간 안에 최소 피드백 수를 보장받는다.

### 초기 규칙 예시

- 피드백 1개 작성: 1 크레딧 획득
- 내 프로젝트 피드백 요청: 3 크레딧 사용
- 보장 조건: 48시간 안에 최소 3개 피드백
- 조건 미달 시: 사용 크레딧 일부 환불 또는 우선순위 상승

### 품질 관리

- 너무 짧은 피드백은 크레딧 지급 제외
- 프로젝트 작성자가 “도움 됨”으로 표시한 피드백은 추가 평판 지급
- 반영된 피드백은 리뷰어 프로필에 기록
- 반복 저품질 리뷰어는 큐 노출 제한

## 9. 프로젝트 상태 모델

각 프로젝트는 상태를 가진다.

- Idea: 아직 만들기 전
- Prototype: 빠르게 만든 초기 버전
- Needs Feedback: 피드백 요청 중
- Iterating: 피드백 기반 수정 중
- Shipped: 공개 출시됨
- Parked: 잠시 보류
- Archived: 더 이상 진행하지 않음

중요한 것은 Archived도 실패가 아니라 기록으로 남기는 것이다. 바이브 코딩은 많이 실험하는 흐름이므로, 접은 프로젝트도 학습 자산이 된다.

## 11. 주요 사용자 흐름

### 프로젝트 등록

1. 사용자가 데모 URL 또는 GitHub URL을 입력한다.
2. 제목, 설명, 썸네일, 사용 도구를 입력한다.
3. 상태를 선택한다.
4. 공개 여부를 선택한다.
5. 프로젝트가 워크스페이스에 추가된다.

### 피드백 요청

1. 프로젝트 상세에서 “피드백 요청”을 누른다.
2. 원하는 피드백 유형을 고른다.
3. 크레딧을 사용한다.
4. 프로젝트가 피드백 큐에 올라간다.
5. 48시간 안에 최소 피드백 수를 받는다.

### 피드백 반영

1. 사용자가 피드백을 읽는다.
2. 각 피드백을 “반영함”, “나중에”, “반영 안 함”으로 표시한다.
3. 변경 사항을 개선 로그로 작성한다.
4. 프로젝트 상태를 Iterating 또는 Shipped로 바꾼다.

## 12. 초기 성공 지표

MVP 단계에서는 가입자 수보다 피드백 루프 완주율이 중요하다.

- 등록된 프로젝트 수
- 프로젝트당 평균 피드백 수
- 피드백 요청 후 48시간 내 충족률
- 작성된 피드백 중 “도움 됨” 비율
- 피드백 반영률
- 프로젝트 상태 변경률
- 사용자가 다시 방문해 프로젝트를 업데이트한 비율
- Archived 또는 Parked 처리된 프로젝트 비율

## 13. 차별화 요약

기존 서비스와의 차이는 다음과 같다.

- Product Hunt: 출시 중심 vs 이 서비스는 진행/개선 중심
- Threads/X: 순간 노출 중심 vs 이 서비스는 프로젝트 히스토리 중심
- 긱뉴스/Reddit: 랜덤 피드백 vs 이 서비스는 보장형 피드백 큐
- Lovable/Replit: 개별 빌더 내부 프로젝트 관리 vs 이 서비스는 여러 도구를 아우르는 메타 워크스페이스
- 일반 포트폴리오: 완성작 중심 vs 이 서비스는 아이디어, 프로토타입, 실패, 보류까지 포함

## 14. 리스크와 대응

### 피드백 품질이 낮아질 수 있음

대응:

- 최소 글자 수보다 구체성 체크를 우선한다.
- 작성자가 도움 됨 표시를 할 수 있게 한다.
- 좋은 리뷰어에게 더 높은 노출과 평판을 준다.

### 피드백 보장이 운영 부담이 될 수 있음

대응:

- 초기에는 크레딧 기반으로 수요와 공급을 맞춘다.
- 미충족 시 환불보다 큐 우선순위 상승을 기본으로 둔다.
- 필요하면 운영자가 큐에 직접 개입할 수 있는 어드민 화면을 둔다.

### 프로젝트가 너무 저품질일 수 있음

대응:

- 프로젝트 상태를 Prototype으로 명확히 표시한다.
- 데모 링크와 설명이 없는 프로젝트는 공개 큐에 올릴 수 없게 한다.
- “로그인 없이 체험 가능”, “모바일 확인됨” 같은 뱃지로 신뢰를 구분한다.

### 단순 디렉토리로 보일 수 있음

대응:

- 첫 화면을 피드가 아니라 워크스페이스로 설계한다.
- 프로젝트 카드에 다음 액션과 피드백 상태를 강조한다.
- 개선 로그와 상태 변화를 핵심 경험으로 만든다.
